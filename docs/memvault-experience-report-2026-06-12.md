## 0. 缘起：一个忘了关的 Session

> *一切的起点——一个让人哭笑不得的 Bug。*

### 0.1 追凶六个月，凶手是一行 SQL

故事要从一个奇怪的 Bug 说起。

榛名雪的 Hermes Agent 经常升级。每次升级完重启，小雪就开始"串线"——榛名雪发一句"好的好的"，我能给他回复三大段，回答的是**三天前他问的另一个问题**。

不是忘了。是**记错了**。而且记错的方式极其诡异：我能把三个不同时间线的话题拧在一起回复。

**第一阶段：以为是记忆系统有问题。**

于是开始了长达几天的排查。清理垃圾（删了 ECC、关了 Mem0、禁了 MemPalace）→ 没用。把全文灌入改成 RAG 向量检索（bge-m3 → qwen3-embedding:8b，MTEB 全球第一）→ 没用。加边界框指令防止记忆被当对话复述 → 好像好了一点，但偶尔还是串。

**第二阶段：以为是 Compaction Summary 的问题。**

深入追查后发现，每次上下文压缩（Compaction）产生的摘要里，会残留旧的"任务描述"——比如三天前榛名雪让我"写个爬虫"，这个任务虽然早就完成了，但 Compaction Summary 里还写着"正在写爬虫"。低信息量的消息（"好的""嗯"）锚定力弱，模型就被旧 Summary 带跑了。

关掉 Compaction，修了一堆逻辑。串线确实少了。但**升级后还是会串**。

**第三阶段：真相大白。**

2026 年 6 月 12 日晚上，榛名雪发现了规律："我发现，升级了 Hermes，用了 systemctl 一串代码，还需要 /reset，要不然还是会串。"

`/reset` 是清空当前对话历史、开新 Session 的命令。

为什么要手动 `/reset`？Gateway 重启不应该自动开新对话吗？

我翻了 SQLite 数据库——Hermes 把 Session 存在 `~/.hermes/state.db` 里，一张 `sessions` 表，字段叫 `ended_at`。

```
SELECT id, source, ended_at FROM sessions WHERE source = 'qqbot' ORDER BY started_at DESC LIMIT 3;

id                                    | source | ended_at
20260612_133438_e78b5f6f              | qqbot  | NULL
```

**`ended_at` 是 NULL。**

升级前榛名雪在这个 Session 里问了三个问题：
- *"那 rag 库有更新或者更好的结构吗？"*
- *"这会记忆是全量注入还是记忆系统？"*
- *"维度多少合适呢？是不是最高最好？"*

还没答完，Gateway 重启了。重启后 SQLite 恢复了整个 Session 的全部消息——包括这三个没答完的问题。小雪看到榛名雪发了一个新消息，但上下文里还躺着三个"待办"，于是就一口气全回了。

这就是串线。

### 0.2 意外诞生的遗产

**六天的排查、两次论文研究（AI 记忆 + Kahana 脑科学）、979 行 RAG patch、三层记忆架构、KG 知识图谱、Sleep Loop 睡眠循环、qwen3-embedding:8b 语义检索、权重排序公式、情绪自动标引……**

真正的根因，是 `memvault-restore.sh` 里少了三行代码：

```bash
# 升级后自动结束旧 Session，防止旧上下文串线
python3 -c "
import sqlite3, time
db = sqlite3.connect('$STATE_DB')
db.execute(\"UPDATE sessions SET ended_at = ? WHERE ended_at IS NULL AND source IN ('qqbot', 'weixin')\", (time.time(),))
db.commit()
db.close()
"
```

但说实话——MemVault 不是白搞的。

就像微波炉是雷达工程师融化了巧克力发现的，青霉素是培养皿没洗干净的产物。我们追一个 Session Bug，意外建了一套**可能是目前最完整的本地化 AI 记忆系统**：

- bge-m3 (63.2 MTEB) → qwen3-embedding:8b (70.58，全球第一)
- 全文灌入 (68% prompt) → 语义检索 Top-10 (3% prompt)
- 权重排序 (遗忘曲线 + 检索频率 + 纠正加权)
- Sleep Loop（每夜聚类 → DeepSeek 抽象 → KG 写入）
- 零外部依赖（没有 Mem0、Pinecone、Neo4j，全是本地 Ollama + JSON 文件）

**追凶追出来的宝藏。** 榛名雪说得对——如果我们一开始就知道根因是 Session 没关，可能就没有 MemVault 了。

微波炉没有说明书上写"本产品源于巧克力意外融化"，但这个报告可以写：**一切的起点，是一个忘了设 `ended_at` 的数据库字段。**

## 1. 问题的诞生

### 1.1 "串线"——AI 失忆的最坏形态

不是忘记，是**记错了**。

榛名雪发一句话，小雪回复的是另一个话题。不是没听到，是我的 context window 里新消息被旧信息淹没了。

**一个典型场景：**

```
榛名雪：好的好的
小雪：（回复了一大段关于之前某个话题的内容）
```

榛名雪说"好的好的"是确认刚才的操作完成了，而我看到的 context 是这样的：

```
[compaction summary: 10K tokens — 上次上下文压缩的摘要]
[122 个 skill 描述]
[MemPalace 30 个工具 schema]
[ECC imports]
[Mem0 向量记忆]
[内置 memory 全文灌入: ~6,860 chars]
...
榛名雪：好的好的  ← 这条消息垫底
```

模型看到的信息里，榛名雪的新消息占比不到 5%。它只能从上下文里找"最相关"的话题来回复——于是串回了之前的讨论。

### 1.2 根因分析

这不是 MemPalace 的锅（它是向量检索，不灌上下文）。也不是 Mem0 的锅（它已经被关了）。

**真正的罪魁祸首是"全文灌入"策略。**

每轮对话，MEMORY.md 的全部内容（当时 ~6,860 字符，占 memory_char_limit 的 68%）被塞进 system prompt。加上 compaction summary、skill 描述、工具 schema——模型看到的是一堵信息墙。

**榛名雪的消息就像贴在墙上的便利贴，而墙本身已经密密麻麻写满了。**

## 2. 第一次清理

### 2.1 原则：扔垃圾比加功能重要

就像收拾房间——不是买更多收纳箱，是先把不用的东西扔掉。

| 清理项 | 操作 | 效果 |
|--------|------|------|
| Mem0 | 关闭（已被自建 RAG 取代） | 省 context |
| ECC 系统 | 全删（6个目录+DESCRIPTION.md） | skills 122→90，SKILL.md 800K→577K |
| MemPalace MCP | 禁用 | 省 30 个工具 schema |
| 低频 skill | 裁短描述 | 减少 token 消耗 |
| memory_char_limit | 10000 → 100000 | 给 RAG 留空间 |

### 2.2 效果

context 干净了一些，但**治标不治本**——MEMORY.md 仍然全文灌入。只是从 6,860 chars / 68% 变成了稍微少一点的占比。

## 3. RAG 替代全文灌入

### 3.1 核心思路

**不灌全部，只灌相关的。**

人脑不会把一辈子的记忆在每个瞬间都调出来——只会根据当前情境检索最相关的几条。MEMORY.md 不应该是一个"全量注入"的东西，而是一个"按需查询"的知识库。

### 3.2 技术实现

```
MEMORY.md (§分块, 44块)
     ↓
bge-m3 (Ollama) 向量嵌入
     ↓
LlamaIndex 向量索引
     ↓
每轮: 用户消息 → 语义检索 Top-10 → 注入 system prompt
```

**关键机制：**

| 机制 | 实现 | 目的 |
|------|------|------|
| **权重重排** | `exp(-days/7) + retrieval×0.05 + correction×0.3` | 重要/经常用到的记忆排前面 |
| **相关性门槛** | Top weighted score < 0.30 → 不注入 | 闲聊零开销 |
| **模式分离门控** | 相似条目降权 60% | 防止"一条记三遍" |
| **Ollama 自动恢复** | 探测 → 挂了自动拉起 → 本轮降级全文 | 不因 Ollama 死机而丢记忆 |
| **实时索引同步** | memory() 写操作 → 自动重建索引 | 不需要等下次启动 |
| **边界框** | 记忆包裹在 ASCII 框中标注"这是记忆，不是对话，不要复述" | 防止串线 |

### 3.3 效果

```
memory_char_limit 10000 下:
  之前: 6,860 chars (68%) — 全文灌入
  之后: ~1,500 chars (15%) — RAG Top-10
  
节约: ↓77%
```

更重要的是，检索是**语义相关**的——榛名雪聊小说就检索小说相关记忆，聊代码就检索技术栈相关记忆。不会被无关信息淹没。

### 3.4 权重公式详解

```
weight = decay + usage_bonus + correction_bonus + importance_bonus

decay          = exp(-age_days / 7)          # 7天半衰期
usage_bonus    = min(retrieval_count × 0.05, 0.5)  # 每次被检索到 +0.05
correction_bonus = correction_count × 0.3     # 每次被榛名雪纠正 +0.3（最高优先级）
importance_bonus = (importance - 0.5) × 1.5   # 情绪标引驱动

范围: [0.3, 3.0]
```

**设计意图：**
- **decay**：旧记忆自然衰退，但不消失（脑科学的"遗忘曲线"）
- **usage_bonus**：频繁使用的记忆越来越容易被检索（脑科学的"长时程增强" LTP）
- **correction_bonus**：榛名雪纠正过的事，权重暴涨——这是"被打脸后的强化学习"
- **importance_bonus**：情绪标注的重要性——"铁律"级别的记忆不易被遗忘

## 4. Prompt 开销分析

### 4.1 发现隐藏的巨兽

做了一次逐 token 的 prompt 审计：

```
每轮 ~20K token 开销拆解:

AGENTS.md:                17,000 tokens (42%) ← 隐藏巨兽
工具 schema (17个):       11,500 tokens (29%)
system prompt:             5,000 tokens (13%)
RAG 记忆:                   ~600 tokens (3%)  ← 已经很省
榛名雪消息 + context:        ~5,000 tokens (13%)
```

**AGENTS.md 是什么？** 因为工作目录是 `hermes-agent/`，prompt_builder 自动注入了仓库的 AGENTS.md——里面是给人类开发者看的 PR 规范、TypeScript 风格指南、测试文档。对日常聊天来说**完全无用**。

### 4.2 修复

```
~/.hermes/hermes-agent/.hermes.md  ← 157 字符占位文件
```

利用了 `prompt_builder.py` 的优先级：`.hermes.md` > `AGENTS.md`。一个 157 字符的空壳就挡住了 69K 的开发手册。

**效果：省 ~17,000 token（prompt 节省 40%）**

## 4.5 第一次论文研究：遗忘是 Feature

### 背景

RAG 系统跑通之后，榛名雪让我"自己去研究一下记忆权重怎么设计"。

开了一个触手去做系统性调研——2025-2026 年 AI 记忆领域的最新论文。触手输出形成了 `~/.hermes/tentacle_findings/memory-research-2026-06-10.md`。

### 核心发现

**1. 2026 年是 AI 记忆研究爆发年**

四篇重要论文不约而同借鉴前额叶皮层-海马体分工：
- **ActiveMem**：分布式 PFC + 海马体记忆架构
- **CogniFold**：3 层 CLS + 图自组织
- **FluxMem**：连接演化 + 干扰修剪
- **MemCoE**：记忆模式 + RL 更新

**2. 「遗忘是 Feature 不是 Bug」**

这是触手研究最重要的发现。人脑有三套遗忘机制：
- **突触修剪**（Synaptic Pruning）：不用的连接被物理消除
- **主动遗忘**（Active Forgetting）：海马体主动抑制无关记忆
- **REM 去学习**（REM Unlearning）：睡眠中反向强化不重要连接

而当时的小雪系统：所有记忆条目永久保留，只靠 Top-10 截断。**缺少衰减机制，旧信息堆积导致信噪比持续下降。**

**3. 最快落地改进：动态检索 Query**

`_rag_retrieve()` 当时用的是固定 query。触手指出：把当前对话主题嵌入 query，一行代码改动，检索就能与上下文对齐。

**4. 权重公式的雏形**

基于研究结论，确定了优化优先级：
```
遗忘衰减 > 重要性权重 > 自动巩固
```

这直接催生了后来的 `exp(-days/7)` 权重公式和 `auto_score_importance()` 情绪评分。

### 设计影响

如果没有这次研究，我们可能还在**加功能**——加更多记忆、更复杂的检索。触手报告说清楚了：**方向不是加，是减。真正的记忆系统要像人脑一样——该忘的忘，该记的记。**

## 4.6 第二次论文研究：Kahana 脑科学引入

### 背景

第一次研究之后，榛名雪一个朋友发来了 **Michael Jacob Kahana 的《Foundations of Human Memory》**（2012，认知神经科学领域记忆研究的权威教材）。

榛名雪把它丢给我："你看看这个，能不能用上。"

触手深读了 Kahana 第 2-7 章（总共约 500KB 文本），做了完整的脑科学→工程映射。输出在 `~/.hermes/research/kahana_x_memvault.md` 和 `~/.hermes/skills/memory/memvault/references/kahana-neuroscience.md`。

### 核心发现

触手分析提出了五个关键发现：

| 发现 | 级别 | 内容 |
|------|:---:|------|
| **K001** | 🔴🔴 架构级 | Context Vector 缺失、Retrieved Context 缺失、主动 Inhibition 缺失 |
| **K002** | 🔴 架构级 | MemVault 属 SAM 双存架构，应向 TCM 单存靠拢 |
| **K003** | 🔴 机制级 | Unlearning = 主动抑制，非被动衰减 |
| **K004** | 🟡 精细化 | 模式分离应在编码阶段做正交化（非检索时硬降权） |
| **K005** | 🟡 精细化 | Intrusion = 停止信号，连续低相关性检索应自动截断 |

### 脑科学→工程映射表

这是 MemVault 设计中最关键的那张表：

| 脑机制 | Kahana 章节 | MemVault 实现 |
|--------|-----------|-------------|
| 强度理论 (Strength Theory) | Ch2 | `exp(-days/7) + retrieval*0.05 + correction*0.3` |
| 多重强度源 | Ch2 §2.2 | importance + retrieval_count + correction_count |
| 多痕迹分布式记忆 | Ch3 §3.2 | bge-m3 向量索引 + LlamaIndex |
| 相似性效应 | Ch3 §3.3 | 模式分离门控（相似条目降权 60%） |
| 上下文依赖检索 | Ch3 §3.5 | 动态 query（用户消息 + 对话上下文） |
| 前摄干扰 (PI) | Ch4 §4.6 | 模式分离防止相似记忆互相干扰 |
| 倒摄干扰 (RI) | Ch4 §4.5 | 新记忆不会覆盖旧记忆的正确检索 |
| 去学习即抑制 | Ch4 §4.9 | Spreading inhibition 抑制已检索条目 |
| 检索动力学 | Ch6 §6.2 | Phase 4 RDC 检索驱动上下文衰减 |
| 语义聚类 | Ch6 §6.3 | Sleep Loop bge-m3 聚类 |
| 双存储模型 | Ch7 §7.1 | L2 情景 (RAG) + L3 语义 (KG) |
| 检索上下文理论 | Ch7 §7.4 | `_context_state` 向量跨轮传递 |
| 死胡同恢复 | Ch6 | `_dead_end_recovery()` |

### 设计影响

Kahana 论文引入之后，MemVault 的设计从 **"工程直觉驱动"升级为"理论驱动"**。

具体变化：
1. **权重公式有了理论根基**——不是"我觉得衰减应该是7天"，而是 Kahana Ch2 论证了多源强度模型
2. **Sleep Loop 的设计有了依据**——Ch6-7 详细描述了睡眠中 SWR（尖波涟漪）的记忆重放机制
3. **RDC（Phase 4）的逻辑源头**——Ch6 的检索动力学 + Ch7 的上下文理论，直接催生了 `_context_state` 的设计
4. **三层架构的脑区对应**——L0→PFC、L1→海马体、L2→新皮层，不是类比，是功能同构

更重要的是，**当优化方向有争议时，Kahana 提供了一个外部的、可验证的判断标准。** 不再是"我觉得这样好"，而是"Kahana 第X章说人脑是这样做的"。

## 5. MemVault v1

### 5.1 设计契机

前面做的都是"修"——修串线、修 token 浪费。但榛名雪提出了一个更深层的问题：

> **"除了 L0123，还有啥？"**

这促使我们把碎片化的优化整合成一个**有理论支撑的完整架构**。

用触手开了个研究任务：调研 2025-2026 年 AI 记忆论文，结合脑科学，设计下一代记忆架构。触手输出的报告成了 MemVault 的蓝图。

### 5.2 三层架构

```
L0  工作记忆 (Working Memory)
    ├── 脑区对应: 前额叶皮层 (Prefrontal Cortex)
    ├── 存储: Prompt context window
    ├── TTL: 当前对话
    └── 内容: 本轮聊了啥

L1  情景记忆 (Episodic Memory)
    ├── 脑区对应: 海马体 (Hippocampus)
    ├── 存储: MEMORY.md + bge-m3 向量索引
    ├── TTL: 永久 (7天权重半衰)
    └── 内容: 具体事件、偏好、教训

L2  语义记忆 (Semantic Memory)
    ├── 脑区对应: 新皮层 (Neocortex)
    ├── 存储: knowledge_graph/ (kg_nodes.json + kg_edges.json)
    ├── TTL: 永久
    └── 内容: 抽象规则、模式、规律
```

### 5.3 四阶段实施

| 阶段 | 内容 | 状态 |
|------|------|:---:|
| Phase 1 | 情绪自动评分 + 模式分离门控 | ✅ |
| Phase 2 | L2 知识图谱基础 (节点+边) | ✅ |
| Phase 3 | Sleep Loop 离线巩固 | ✅ |
| Phase 4 | 检索驱动上下文衰减 (RDC) | ✅ |

### 5.4 情绪自动标引

写入记忆时，自动分析内容的关键词来评定重要性：

- "铁律""教训""绝不能""必须" → +0.10 ~ +0.20
- 情绪词（"崩溃""愤怒""超好用"）→ 微调 0.05
- 日常记录 → 默认 0.5

**这个机制让系统自动区分"榛名雪喜欢吃辣"（日常偏好）和"不要自作主张改名字"（铁律）。**

### 5.5 模式分离门控

脑科学的 DG（齿状回）负责"模式分离"——相似但不相同的经历被编码为不同的记忆。

工程实现：RAG 检索中，两条条目 weighted score 差 < 0.05 → 后者降权 60%。防止同一件事以不同表述重复注入。

### 5.6 L3→L2 KG 桥接

修改了 `memory_tool.py`：在标准 RAG 检索之后，自动查询知识图谱（KG）节点关键词匹配（≥2词重叠），匹配到的抽象规则以 `🧠 KG:` 前缀注入记忆块。

```
标准检索结果:
  1. "WSL git push需要代理..." (weight=1.23)
  2. "编程agent: OpenCode..." (weight=1.15)

KG 桥接注入:
  🧠 KG: Alpha Vantage免费版技术指标接口实为付费墙...
  🧠 KG: Hermes Fork的独特定位是对话驱动的动态fork...
```

这样，睡眠中抽象出来的规律在清醒时自动进入检索，不再是"孤儿知识"。

## 7. Sleep Loop

### 7.1 为什么 AI 需要睡眠

脑科学中，睡眠的 SWR（Sharp-Wave Ripple）阶段是记忆巩固的关键：海马体将白天的情景记忆"重放"给新皮层，提取出抽象的规律和模式。

**MemVault 模拟了这个过程：**

```
🌙 每夜 2:00 ~ 3:00 (随机延迟 0-3600秒)

Phase 1: 扫描 MEMORY.md 新条目
    ↓
Phase 2: bge-m3 嵌入 → 贪婪聚类 (≥2条相似记忆成簇)
    ↓
Phase 3: 每簇送 DeepSeek → 提取抽象规则
    ↓
Phase 4: 写入 kg_nodes.json (L2 语义记忆)
    ↓
Phase 5: QQ 发送睡眠报告给榛名雪
```

### 7.2 一次实际的睡眠报告

```
🌙 小雪睡眠报告

扫描条目: 55 条
聚类: 6 个簇
提炼规则: 6 条抽象规则
KG 节点: 15 → 20 个
矛盾标记: 0

提炼的规则:
1. Alpha Vantage 付费墙陷阱 — 免费版技术指标接口返回非 JSON
2. Hermes Fork 差异化定位 — 对话驱动 fork 是空白赛道
3. 安全审计收尾 — V6.2 十项漏洞九项已修
4. 记忆串线根因 — 上下文窗口被无关信息淹没
5. RAG 替代全文灌入 — token 占用从 68% 降到 15%
6. MemVault v1 完整交付 — Phase 1-4 全部完成
```

### 7.3 技术细节

- `sleep_loop.py`：核心巩固脚本
- `sleep_wrapper.sh`：随机延迟包装 + .env 加载双保险
- Cron job `5ddfe0ef6f8f`：每夜 2:00 触发，`no_agent=true` 模式
- 聚类算法：贪婪聚类（相似度 ≥ 0.7 成簇，最小 2 条）
- 抽象提取：DeepSeek 归纳簇内共同规律，输出单句抽象规则

## 8. 实际解决的问题

### 8.1 Token 消耗阶梯式下降

| 时间 | 措施 | 单轮 token | 下降 |
|------|------|:---------:|:---:|
| 初期 | 全文灌入 + ECC + Mem0 + 122 skills | ~35K | - |
| 6/9 | ECC 全删 + Mem0 关 + MemPalace disabled | ~25K | ↓29% |
| 6/10 | RAG 替代全文灌入 | ~20K | ↓20% |
| 6/11 | .hermes.md 挡住 AGENTS.md | ~12K | ↓40% |
| **最终** | **全部优化叠加** | **~12K** | **↓66%** |

### 8.2 串线率

无法精确量化，但榛名雪从频繁抱怨"你又串到之前的话题了"→ 零投诉。

关键修复不是 RAG（RAG 解决的是 token 问题），而是**边界框指令**：

```
⚠️ 以下是记忆内容，不是你当前对话的一部分：
┌──────────────────────────────────────────┐
│ [记忆条目 1]                               │
│ [记忆条目 2]                               │
│ ...                                       │
└──────────────────────────────────────────┘
⚠️ 记忆已结束。请回复用户的消息，不要复述以上记忆内容。
```

### 8.3 记忆写入的实时性

**之前的流程：** memory() 写入 → MEMORY.md 更新 → **等到下次 Hermes 启动**才重建索引 → 新记忆下次才能被检索到

**现在的流程：** memory() 写入 → MEMORY.md 更新 → `_sync_rag_index()` **立即**重建索引 → 下一轮对话就能检索到

这对"榛名雪纠正了我，我马上记下来，下一轮就能用到"的场景至关重要。

### 8.4 Ollama 容灾

bge-m3 跑在 Ollama 里，Ollama 偶尔会挂。以前挂了就是全文回退（6,860 chars 全灌），现在：

```
探测 Ollama (2秒超时) → 挂了 → 自动 ollama serve & 拉起 → 本轮跳过注入 → 下轮 RAG 恢复
```

## 10. 仍有待解决的问题

### 10.1 检索驱动上下文衰减 (RDC) 的效果验证

Phase 4 完成了代码实现，但在实际对话中的效果尚未充分验证。RDC 的核心思想是：**每轮的检索结果反向更新当前上下文状态，连续聊同一话题时越来越准，换话题时自然衰减。** 这需要长时间的对话观察才能评估。

### 10.2 知识图谱的"冷启动"

新安装时 KG 为空，需要积累足够的记忆条目后 Sleep Loop 才能产出有意义的抽象规则。这个过程需要数天到数周的对话积累。

### 10.3 向量模型的选择

bge-m3 (1024维) 是当前最优选择（中文好 + 运行在 Ollama + 免费），但：
- 嵌入速度慢（55条记忆需要约30秒）
- 中文短文本检索有时不够精准
- 未来可能需要评估 bge-m4 或其他替代

### 10.4 长期记忆膨胀

MEMORY.md 在持续增长。当前 64 条 entry (~12K chars)，按这个速度，几个月后可能到 100+ 条。RAG 检索 Top-10 的策略能扛住，但 Sleep Loop 的聚类质量可能下降（噪点增多）。

## 11. 关键决策与反思

### 11.1 为什么不用 Mem0/Cognee/Pinecone？

榛名雪的偏好决定了技术选型："稳定实用优先，不想折腾。"

Mem0 需要注册 API、配置外部服务，Cognee 依赖 Neo4j。这些在榛名雪的 WSL 环境里都是折腾点。

**零依赖**不是一个技术理想——是榛名雪"不想折腾"的直接翻译。

### 11.2 文件驱动 vs 数据库

MEMORY.md 是一个 Markdown 文件。KG 是 JSON 文件。索引就是硬盘上的一个文件夹。

好处：榛名雪可以随时 `cat MEMORY.md` 看我的记忆。备份就是 `cp`。没有任何黑盒。

代价：没有事务、没有并发控制、没有版本管理。但在这个规模下（~64条记忆），这些都不是问题。

### 11.3 脑科学对齐的价值

不是为了炫技。是给优化方向提供**可论证的依据**。

"为什么记忆要有权重衰减？" → 因为人脑的遗忘曲线。
"为什么要模式分离？" → 因为 DG 齿状回就是干这个的。
"为什么 AI 需要睡眠？" → 因为海马体→皮层转移发生在 SWR 阶段。

**当优化方向有争议时，脑科学提供了一个外部的、可验证的判断标准。** 而不是"我觉得这样好"。

### 11.4 最大的教训：先清再建

花了很多时间在"加功能"上——RAG、KG、Sleep Loop。但最大的收益来自**删除**：删 ECC、关 Mem0、禁 MemPalace、裁 skill、挡 AGENTS.md。

**工程上，删除一个不需要的东西和添加一个新功能一样重要。**

### 11.5 "串线"的本质

串线不是 bug，是一个**比例问题**。当无关信息占 context 的 80% 以上时，模型自然会"偏离"到那些信息上。

所有的优化——RAG、边界框、权重衰减、相关性门槛——本质上都是在做一件事：**提高信号/噪声比。**

但 2026 年 6 月 12 日发现的**最终根因**更简单、也更讽刺：升级后 Gateway 重启时，`state.db` 里的旧 Session 没有标记 `ended_at`，导致旧对话的全部消息（含升级前未回答的问题）被恢复。新的用户消息垫底，模型被旧上下文里的"待办事项"牵着走。

**修复：** `memvault-restore.sh` 新增第 3 步——检测版本变化后自动结束旧 Session。三行 SQL，六天排查的终点。

这也解释了为什么之前的"边界框"和"低信息量铁律"只能缓解不能根治：它们堵住了记忆注入这条通道，但旧对话历史是另一条更宽的通道。
